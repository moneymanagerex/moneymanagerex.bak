Introduction

Custom Reports allows the user to create unique reports using an SQL script or a Lua script. Lua is a complete language in itself, and has been incorporated into MMEX to enable the user to write a more comprehensive report to suit the user. The Custom Reports Manager is used to install or create an SQL or Lua  script, that can communicate directly with the database. The report script is added to the navigation tree of MMEX using a Script Heading. A Heading can also be added without an associated script, to allow scripts to be set up in sections. The Custom Reports Manager can also be used to run the scripts for testing purposes.

The Custom Reports Manager Interface has the following features:

Report Title
Text field

This is used to give the script a Title.
The title is used to access the script from the navigation tree.

Heading
Check Box

A Heading can be added without an associated script.
A Menu item has an associated script.

When Checked: Allows a title to be used as a Heading in the navigation tree.

Sub-Menu
Check Box

When Checked: The navigation tree will set the title as a sub-menu under the previous menu item. The menu item is not necessarily a heading.

SQL Source Script Text Field

This is where the script details can be written.

Open Button

Load an existing script from a file.

Save Button

Allows the user to save the SQL script to the existing file or a new file.
The Report Title will be used as the file name. Spaces in the title will be changed to underscore characters in the file name.

Run Button

Allows the script to be executed without closing the dialog.

Warning: Script changes will be lost when script errors occur.
Recommendation: Save the script before using Run

Clear Button

Clears the SQL Source Script area only.

Close Button

Closes the dialog without saving.
A way of cancelling out of the dialog.

Once the script is created, and saved in MMEX, the report can be easily accessed via the Navigation Tree.

The creation of a new Custom Report is via:

·         The menu item: Tools –> Custom Reports

·         The icon on the toolbar.

This action will open the Custom Reports Manager dialog, and allows the user to directly write a script and save it as a file.

Useful Custom Scripts are also available from the MMEX Forum on the Internet, which can be accessed as follows:

·         Use the menu option: Help –> Report Issues or Feedback

·         Search the forum heading: Custom SQL Report Scripts

Recommendations:

·         For new scripts, use a useful title for your report.

·         Save the script before running in the Custom Reports Manager.
Note: Script errors may result in loss of your script if not saved first.

·         Use the pop-up menu on the navigation tree to edit the file if necessary.

Note:

·         All script titles are saved in an index file called: CustomSQLReportIndex.txt

·         Custom SQL Reports can be removed by deleting the index file.

·         Reports can be reorganized by directly modifying the index file if necessary.

Running Custom Reports

Reports are designed to run from the MMEX Navigation panel.

Reports can also be run via the Custom Reports Manager if the user choses to modify the script. The Custom Reports Manager also allows other functions.

The Custom Reports Manager will allow the user to modify, install or create new reports that will be permanently recorded in the MMEX Navigation Tree. The report name is set when the report is saved. By using the report title, the user can:

·         Run the report

·         Edit the report

·         Delete the report.

When a report title is saved without an associated script file attached, the title can be used as a section heading, to group scripts together. The MMEX Navigation tree then behaves similar to the standard reports.

Reports can only be nested one level deep.

Table View: AllData

MMEX creates a special view called AllData. This view can easily be used by SQL Scripts to create specialized reports that users may require without having to create complicated scripts.
For more information use the SQL Command: select * from AllData;

For closer inspection, a script file can be created to view the data in each table using the following SQL Command:
select * from <table_Name>
where <table_name> is one of the table names above.

Table Views

SQL has the ability of creating virtual table definitions from existing tables. These virtual tables, called Views, use the data from the original base tables, and can be used in reports like tables. The SQL command for creating a view is:

create view <VIEW_NAME> as <SQL STATEMENT>;

Where:
<VIEW_NAME> is the name of the view.
<SQL_STATEMENT> is the SQL statement that defines the view.

Example:
create view budget as
select * from BudgetYear_V1 BY, BudgetTable_V1 BT, Category_V1 C, SubCategory_V1 SC
where BY.BudgetYearID = BT.BudgetYearID and BT.CategID = C.CategID and BT.SubCategID = SC.SubCategID;

To inspect the view, use the SQL command: select * from Budget;
To remove the view use the SQL command: drop view Budget;

SQL Scripting (Introduction)

Attempting to teach SQL programming is beyond the scope of this help file, but the following is a quick guide into understanding scripts, and possibly creating your own scripts if required.

In creating scripts, the main command we would tend to use is the SELECT statement.

The basic format is:

SELECT <name>

<name> represents the field names of the table, or * for all names.

FROM <table>

<table> represents the table or tables the command is aimed at.

WHERE <condition is met>

[optional] here we test for some condition.

ORDER BY <parameter>

[optional] sorts the output by this parameter


Examples:

SELECT * FROM AccountList_V1;
* This would display all fields for the table: AccountList_V1

SELECT AccountName, Status FROM AccountList_V1 where AccountType = "Term";
* This would display the Term Account names and the account status from table: AccountList_V1

Here we describe a more complex script:

Although this script is useless, it demonstrates some basic principles.

select pT.PayeeName payee,
    cT.CategName,
    scT.SubCategName
from payee_v1 pT,
    category_v1 cT,
    subcategory_v1 scT
where
    pT.categID = cT.categID and pT.subCategID = scT.subCategID
order by
    payee


In this example we see the table name is: payee_V1 and the field name is PayeeName

In the FROM clause we see the table name: payee_V1 pT
* Here the pT at the end, acts as an alias for the table name.

Now in the SELECT we see: pT.PayeeName payee
* The pT. is used to state the table that owns the field PayeeName
* The payee at the end, is an alias for the table.fieldname combination

We can separate table names and field names with commas, as we see more tables and fields in this example.

In the WHERE clause, we see: pT.categID = cT.categID
* Here we test to make sure the two fields are the same in both tables.

Now in the ORDER BY clause, we can now use the simplified name of payee we defined in the SELECT clause.

I hope this short introduction takes the mystery out of the script files as seen in the MMEX Forum in the Custom SQL Report Scripts section.

Brief Introduction to Lua

Lua is a programming language, and as such has a lot of similarities to other programming languages.

Data is stored in a variable which is given a name, and we assign some data to it.

Suppose we want to store a name we write:

table_name = "category"
Now every time we want to use the name category, we use the name table_name


When we write code, we generally have a Start, and an End point.
Start Point:

table_name = "category"

The first statement encountered, which can be resolved.


End Point:

return table_name

This is the return statement.
Here we return a value to the calling program, which for our case is MMEX.


Returning just the table_name in a lot of circumstances is fine, but for an ending, not so good.
To combining data to have meaning, Lua introduces the dot dot notation.

return "Real table name is " .. table_name .. "_v1"

Combines the data to return one value as:
Real table name is category_v1


For a few lines of code, this is fine, but to make coding easier, we use Functions. We can think of a function as a building block, which can be used many times. We can change the output by giving the function different values. Functions may or may not return a value.

Note:
A function cannot be executed as a starting point for a main program.
This is because a Function cannot be resolved.

Defining a function:

function sql_statement(table_name)



end

Here we declare a function as sql_statement and we give it the name of the table.

Note: Because a function is a block of code, it starts with the word function and ends with the word end

Purpose of the function:

function sql_statement(table_name)
    local sql_code = "select * from " .. table_name .. "_v1"
    return sql_code
end

Now every time we use this function we get the sql statement for that table.
Makes life much easier.

Scope of Variables
A variable can be defined at any point, and immediately has global access.
A local variable is only available within a block of code.

Global Access

Local Access

function GetTableName()
    table_name = “payee_v1”
end

function GetTableName()
    local table_name = “payee_v1”
end

-- Main (Starting point)
    GetTableName()
    my_table = table_name
return my_table

-- Main (Starting point)
    GetTableName()
    my_table = table_name
return my_table

This code will display: payee_v1

This code will display nothing.

Arrays: This is where we can assign our values in a block, known as an array

selectable_tables = {"Payee","Category","SubCategory"}

Here we grouped these as a single block called:
selectable_tables



Above we defined our own function, but Lua has other functions provided for the user to use. Combining our functions with the inbuilt functions we can produce the following program which is moderately useful:

    table_name = mmGetSingleChoice("Select a table", "Lua", selectable_tables)
    sql_script = sql_statement(table_name)
    value = mmGetSQLResultSet(sql_script)
return value

Get the table name.
Get the sql code for the table
Get the table data
return the result for MMEX to display

Using our function to generate the sql code made programming easier, and also easier to maintain.

While this is a trivial programming exercise, it provides a brief introduction into programming, using Lua as the programming language.
